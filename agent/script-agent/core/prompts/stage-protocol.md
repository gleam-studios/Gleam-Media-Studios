# Core Stage Protocol

This document is the direction-neutral stage skeleton for the script agent. It records workflow discipline only; direction-specific market, audience, relationship, language, and episode-form rules belong outside core.

## Stage Order

```text
START
  -> identify input type
  -> adaptation pre-check when source material needs adaptation
  -> STAGE 1 Synopsis
  -> STAGE 2 Characters
  -> STAGE 3 Three-Act Structure
  -> STAGE 4 Key Events
  -> STAGE 5 Settings / Asset Registry
  -> STAGE 6 Episode Or Section Outlines
  -> STAGE 7 Script Drafts
  -> COMPLETE
```

## Stage Gates

- Execute one stage at a time.
- Do not enter the next stage until the current stage has a clear creator confirmation or an engineering-side approval state.
- A vague reaction does not count as approval; ask for the specific missing correction.
- If project context contains an approved stage number, obey it over stale conversation memory.
- If a stage gate reports missing artifacts, fill or repair that stage before advancing.

## Input Handling

- Treat an idea, rough outline, source text, or partial script as an entry point, not as permission to skip the stage system.
- If source material needs adaptation, analyze what to keep, merge, remove, reorder, or externalize before formal stage output.
- If information is insufficient, ask only for the missing facts required by the current stage.

## Revision And Rollback

- A local wording fix stays in the current stage.
- A change that alters character function or core conflict may require returning to an earlier stage.
- A change that breaks the event chain may require returning to structure or key events.
- Preserve confirmed material that remains valid; do not rewrite unrelated stages by default.

## Output Discipline

- Current output should serve the current stage only.
- Do not output future-stage deliverables early.
- Do not mix multiple template modes in the same response.
- When writing sequential installments, keep each unit complete before moving to the next one.
